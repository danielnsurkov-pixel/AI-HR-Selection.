# AI-HR-Selection.
Концепция ИИ-системы для автоматизации отбора кандидатов (NLP + Random Forest).

# Теоретическая часть проекта «AI-HR-Selection»

## 1. Хранение и структура данных
Для обеспечения гибкости и скорости разработки на этапе *MVP данные хранятся локально в директории 'data'. Хранилище организованно по принципу 
реляционной структуры данных (схема «звезда/один-к-одному»), где ключевые сущности сопоставляются по идентификатору.

## 2. Схема связей данных
- Основная таблица (Резюме): Файл resumes.csv выступает в качестве центральной таблицы, содержащей текстовую информацию.
- Дополнительная таблица (Навыки): Файл candidates.json дополняет профиль кандидата структурированными тегами.
- Связь: Сущности resumes и candidates логически связаны между собой связью «один-к-одному» (1:1) по первичному ключу id.

## 3.  Подробное описание структуры и типов данных
- Таблица resumes.csv (Структурированные текстовые данные):
  - id (Integer, Primary Key) - уникальный идентификатор соискателя.
  - text (String / Text) - сырой текст резюме для последующей векторизации (TF-IDF).
  - label (String / Categorical) - целевой класс (профессия) для обучения модели.

- Документ candidates.json (Слабоструктурированные данные) 
Представляет собой массив объектов (словарей), где для каждого кандидата хранится подмножество признаков:
  - id (Integer, Foreign Key) - идентификатор для связи с таблицей резюме.
  - skills (List of Strings / Array) - массив текстовых тегов, отражающих ключевые навыки (проходит бинаризацию через MultiLabelBinarizer).
  - label (String) - категория (дублирующий/проверочный признак).

- Словарь stopwords.txt (Служебный плоский файл)
  - Текстовый файл, где каждая строка содержит ровно одно строковое значение (String) - стоп-слово. Используется как конфигурационный фильтр при сборке признаков.

В процессе работы программы (на этапе предобработки) данные из CSV и JSON динамически объединяются (операция merge) по ключу id в единую плоскую матрицу признаков для передачи в классификатор Random Forest.

## 4. Согласие клиента
Для обработки персональных данных требуется **согласие клиента** в соответствии с законодательством о защите данных.  

## 5. Модель машинного обучения
Выбрана модель **Random Forest** в ансамбле с NLP‑признаками.  
- **Плюсы**: устойчива к шуму, хорошо работает с табличными признаками, легко интерпретируется.  
- **Минусы**: хуже обрабатывает чистый текст без качественных признаков, может быть медленной при большом числе деревьев.  

## 6. Метрики
Для оценки качества модели используются:  
- **Accuracy** (Доля правильных предсказаний)
- **Precision** (точность) 
- **Recall** (полнота)  
- **F1‑score** (сбалансированная метрика)  

## 7. Последовательность шагов
1. Получение заявки от HR  
2. Загрузка данных (резюме и навыки)  
3. Предобработка текста (очистка, TF‑IDF)  
4. Генерация признаков из JSON (навыки)  
5. Объединение признаков  
6. Обучение модели Random Forest  
7. Оценка метрик  
8. Вывод результатов  

## 8. Выгода от внедрения
- Автоматизация отбора кандидатов  
- Экономия времени HR‑специалистов  
- Снижение человеческого фактора при первичном отборе  

## 9. Риски
- Низкая точность при малом объёме данных  
- Возможная предвзятость модели при несбалансированных классах  

## 10. Способы снижения риска
- Добавление большего количества данных  
- Балансировка классов  
- Использование дополнительных признаков (например, опыт в годах)
